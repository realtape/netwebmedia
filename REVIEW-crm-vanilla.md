# CRM Vanilla Code Review — 2026-05-05

Scope: PHP API in `crm-vanilla/api/` (handlers + libs + index.php + config.php), front-end JS in `crm-vanilla/js/`, schema migrations, and `.htaccess`. Time-boxed at ~15 minutes; sampled the highest-risk modules rather than enumerating every file.

## Verified clean (or acceptable)

- **Schema migrations** — grepped all `schema_*.sql` for the forbidden `SET @x := (SELECT … FROM information_schema)` + `PREPARE/EXECUTE` pattern and for `PREPARE … FROM`; zero hits. Naive `;`-splitter risk also looks clean from a sample read.
- **CSRF / session hygiene** — `index.php` enforces same-origin `Origin`/`Referer` for state-changing methods on non-public, non-token routes. `_guard_session_start()` sets `secure`, `httponly`, `samesite=Strict`. `auth.php` calls `session_regenerate_id(true)` post-login. Token equality uses `hash_equals()` everywhere I sampled (`migrate.php`, `cron_workflows.php`, `domain_audit.php`, `import_csv.php`).
- **SSRF** — `analyze.php` and `proposal.php` go through `url_guard_or_fail()` and pin `CURLOPT_PROTOCOLS` to http/https. (Caveat: see F-04 below.)
- **Prepared statements** — every SELECT/UPDATE/DELETE I read uses parameterised queries; no string concatenation of `$_GET`/`$_POST` into SQL.
- **`crm-vanilla/.htaccess`** — denies `.sql`/`.md`, blocks `api/migrations/` and `api/lib/` from public access.

## Findings

| ID | Severity | File:line | Finding | Recommended fix |
|----|----------|-----------|---------|-----------------|
| F-01 | Critical | `crm-vanilla/api/lib/url_guard.php:57` | DNS-rebinding TOCTOU. `url_guard_or_fail()` resolves the host with `gethostbynamel()` and returns the original URL string — curl then re-resolves and may receive a different (private) IP. An attacker controlling DNS can pass the guard with a public A record and serve `127.0.0.1` to curl on the second lookup. | Resolve the IP once, then either pass the IP via `CURLOPT_RESOLVE`, or set `CURLOPT_DNS_USE_GLOBAL_CACHE=false` and re-validate the resolved peer IP via `curl_getinfo($ch, CURLINFO_PRIMARY_IP)` against the blocklist; reject on mismatch. |
| F-02 | Critical | `crm-vanilla/api/lib/guard.php:73-76` | `require_guard()` returns a synthetic guest record `['id'=>0,'role'=>'guest','status'=>'demo']` for any unauthenticated request, "letting them through". Combined with `index.php`'s `$public_routes` allowlist, this means every non-public, non-write GET handler executes for an unauthenticated caller — they just see rows whose `user_id IS NULL` (legacy seed) and `organization_id IS NULL`. That is intentional per the comment, but it bypasses the documented payment gate for any route that doesn't independently re-check auth (e.g. `admin.php` re-checks; many others don't). | Either (a) remove the guest fall-through and require an authenticated session for all non-public routes, or (b) audit every handler to confirm none leak tenant-scoped data when called as guest. At minimum, document this on every handler that relies on the legacy NULL-rowset behaviour. |
| F-03 | Critical | `crm-vanilla/api/handlers/templates.php:80-117` | POST does not call `require_org_access_for_write('member')`. An authenticated user with `X-Org-Slug: <other-org>` will have `current_org_id()` resolve to that other org, and the INSERT writes `organization_id = <other-org>`. CLAUDE.md explicitly requires this guard on every owned-table INSERT. | Add `require_org_access_for_write('member');` at the top of the POST branch (line 81), matching the pattern used in `contacts.php:63` / `deals.php:54`. |
| F-04 | Critical | `crm-vanilla/api/handlers/social.php:122-155`, `settings.php`, `niche_metrics.php`, `reports.php`, `intake.php`, `seed_templates.php`, `seed_client_templates.php` | Same H2 cross-org INSERT issue as F-03. Grep for files that have `INSERT INTO` but no `require_org_access_for_write` shows ~10 handlers missing the call. Sampled `social.php` (lines 138-150, 303-315) — both insert `organization_id = $orgId` derived from `current_org_id()` which honors `X-Org-Slug`, with no membership check. | Add `require_org_access_for_write('member')` (or `'admin'` as appropriate) before each INSERT. Audit the full list: workflows, wa_flush, ai_draft_reply, deals, contacts, ig_publish, payments, campaigns, conversations, events have it; social, settings, niche_metrics, niche_config, reports, intake, leads (public, OK), templates, seed_*, organizations, import_clickers do NOT. |
| F-05 | Critical | `crm-vanilla/js/marketing.js:212` | Stored XSS. `EMAIL_CAMPAIGNS` is loaded from the API; `c.name`, `c.date` (lines 212, 217) are interpolated into `innerHTML` with no escape. A campaign created by anyone in the org with `<img src=x onerror=...>` in the name will fire on every CRM user who opens Marketing → Email Campaigns. | Wrap with `escHtml(c.name)`, `escHtml(c.date)` (helper already defined and used for templates on line 255). Same pattern likely affects other tables in this file — sweep the function. |
| F-06 | Warning | `crm-vanilla/js/contacts.js:123` | `'Error loading contacts: ' + e.message` written to `tbody.innerHTML` without escaping. `e.message` typically comes from `fetch`/JSON parse, but a hostile API response can place `<script>` markup there. Low exploitability but trivial fix. | `tbody.innerHTML = '<tr><td colspan="7">Error: ' + esc(e.message) + '</td></tr>';` or use `textContent`. |
| F-07 | Warning | `crm-vanilla/js/pipeline.js:871` | `c.source` is rendered raw inside `<span class="deal-card-tag">` when not in the known `srcLabels` map. `source` is an API string — a contact with `source: '<img onerror>'` produces stored XSS in the kanban. | `(srcLabels[c.source] || escapeHtml(c.source))`. |
| F-08 | Warning | `crm-vanilla/api/handlers/track.php:76-111` | Open-redirect chained on tracking token. A valid `t=TOKEN` plus arbitrary `u=https://attacker.com` produces a 302 from netwebmedia.com. Tokens are unique-per-send but the URL parameter is unrestricted — useful for phishing because the visible domain in the email link is `netwebmedia.com/api/?r=track&...`. | Either (a) bind the destination URL to the campaign at send time and look it up in `campaign_sends` instead of trusting `$_GET['u']`, or (b) restrict redirect targets to a whitelist (NWM domains + customer domains registered in `organizations`). |
| F-09 | Warning | `crm-vanilla/api/handlers/conversations.php:46-69` | POST accepts an arbitrary `contact_id` from request body without verifying that contact belongs to the active org. Combined with the GET endpoint's `LEFT JOIN contacts c ON conv.contact_id = c.id` returning `c.name as contact_name`, an authenticated user can create a conversation referencing a foreign contact and read back that contact's name + avatar via the listing endpoint. | After validating `data['contact_id']`, look the contact up under `tenancy_where()` and 404 if not visible to the caller. |
| F-10 | Warning | `crm-vanilla/api/handlers/social.php:288-301` and 322-335 | The `CREATE TABLE IF NOT EXISTS social_scheduled` runtime DDL omits the `organization_id` column, but the INSERT at line 305 references it. On a fresh DB or an environment where the migration hasn't run, the first call creates the table without the column, then every INSERT fails with a SQL error. | Add `organization_id INT UNSIGNED DEFAULT NULL` (and an index) to all three inline `CREATE TABLE` blocks, or remove the inline DDL and rely on a `schema_*.sql` migration. |
| F-11 | Warning | `crm-vanilla/api/handlers/social.php:19`, `crm-vanilla/api/handlers/payments.php:23` | `static $social_ready = false;` / `static $payments_ready = false;` declared at file scope. PHP `static` outside a function is a no-op — this is a regular local each request, not a per-worker cache. The `try { SELECT 1 FROM …; }` runs on every request, defeating the documented intent. | Move the readiness check into a function with a real `static` cache, or use `apcu_*` if APCu is available. |
| F-12 | Warning | `crm-vanilla/api/handlers/migrate.php:27` | Statement splitter explodes on raw `;` after stripping `--` comments only. Inline `/* … */` comments containing `;` and any string literal containing `;` would be torn. Also matches CLAUDE.md's noted limitation. | Either upgrade to a proper SQL splitter, or document at top of every `schema_*.sql` "no semicolons inside string literals or block comments". |
| F-13 | Warning | `crm-vanilla/api/handlers/wa_flush.php:38-41` | Role check uses `$user['role']` directly (the user-table column), not `org_role(current_org_id())`. A master-org owner/admin who is operating in a sub-org context would be blocked even though `org_role()` makes them a virtual admin everywhere. Mirror the pattern of `require_org_access`. | Replace with `$role = org_role(current_org_id() ?? ORG_MASTER_ID); if (!in_array($role, ['admin','owner'], true)) jsonError(...);`. |
| F-14 | Warning | `crm-vanilla/api/config.php:27-32` | Six committed `*_TOKEN` defaults with the literal string `_ROTATED_<hex>`. Comments correctly say "MUST NOT be relied on in production"; meanwhile `cron_workflows.php:11` also hardcodes the string in a doc comment. If `config.local.php` deploy ever fails (which has happened — the file itself documents the 2026-04-29 leak) these get accepted, and the strings are now public via this commit. | Make the constants **fail-closed** when `config.local.php` is missing. e.g. `if (!defined('MIGRATE_TOKEN')) define('MIGRATE_TOKEN', bin2hex(random_bytes(16))); // unrecoverable per-process default — forces config.local.php`. Remove the hex strings from every doc comment too. |
| F-15 | Warning | `crm-vanilla/api/handlers/track.php:62-67`, `83-86` | `UPDATE campaign_sends … WHERE token=?` is **not** scoped by `organization_id`. A token collision (or a token leak across orgs) lets one org's open/click flip another org's send rows. `track_org_id_for_token()` exists and is used for the unsub paths but not the open/click paths. | Pull `organization_id` once per request via `track_org_id_for_token()` and add `AND organization_id = ?` (or `IS NULL`) to every UPDATE. |
| F-16 | Info | `crm-vanilla/api/index.php:11-14` | CORS sends `Access-Control-Allow-Origin: <ALLOWED_ORIGIN>` but does NOT include `Vary: Origin`. With multiple sub-domains potentially calling this API (whitelabel), caches/CDNs may serve the wrong CORS header. Also `Access-Control-Allow-Credentials` is missing — session cookies won't be sent on cross-origin requests, which is fine for this design but worth noting if mobile/Capacitor calls fail. | Add `header('Vary: Origin');`. Verify the ALLOWED_ORIGIN value matches the calling origin (mobile app uses Capacitor's `capacitor://` scheme — won't match `https://netwebmedia.com`). |
| F-17 | Info | `crm-vanilla/api/handlers/intake.php:34-49` | Heuristic-fallback path renders the user's `first_name` directly into the `reply` string returned by the API. If a downstream consumer renders that reply unescaped, this becomes reflected XSS. Low-likelihood but the data flows through workflow templates. | `htmlspecialchars($lead['first_name'] ?? | F-18 | Info | `crm-vanilla/api/handlers/contacts.php:25-37` | `?status=` and `?segment=` filter values are passed straight from query string into LIKE/`= ?`. They're parameterised so no SQL injection, but neither is validated against an enum, so the filter accepts arbitrary strings (returns empty result rather than erroring). Minor data-quality issue. | Validate against `$ALLOWED_STATUSES` / `$ALLOWED_SEGMENTS` and 400 on unknown values. |
| F-19 | Info | `crm-vanilla/js/marketing.js:233, 245` | Inline `onclick="alert('Template builder coming soon.')"` and an `onclick` handler embedding nested single quotes. Under the project's live CSP (per CLAUDE.md, since 2026-04-28), inline event handlers are typically blocked unless `'unsafe-inline'` is allowed in `script-src` — these will silently fail. | Replace with addEventListener bindings, or verify CSP allows them. |
| F-20 | Info | `crm-vanilla/api/handlers/migrate.php:41-52` | The idempotency-error matcher uses `strpos($msg, $code) !== false` — code `'1062'` will match inside any error message that happens to contain those four digits (e.g. row counts, IDs in error messages). Low-impact false positive for now. | Match against `$e->getCode()` (PDO error code) and `$e->errorInfo[1]` (driver code) directly, not substring of `$e->getMessage()`. |

## Summary

20 findings: **5 Critical, 10 Warning, 5 Info.** Top three issues:

1. **F-01 DNS-rebinding hole in `url_guard.php`** — the SSRF guard validates the resolved IP at parse time but curl re-resolves at fetch time. Defeats the entire SSRF defence layer relied on by `analyze.php` and `proposal.php`.
2. **F-03/F-04 cross-org INSERT regression** — at least ~10 handlers (templates, social, settings, niche_metrics, reports, seed_*, organizations, etc.) skip `require_org_access_for_write()`. An authenticated sub-account user can use `X-Org-Slug` to write into a sibling org. CLAUDE.md flags this exact pattern as the "H2" finding the writes are supposed to defend against.
3. **F-05 stored XSS in marketing.js Email Campaigns table** — campaign names rendered raw into `innerHTML`. Most other tables in the same file already use `escHtml`; this one was missed.

Other notable items: F-02 (guest fall-through in `require_guard()`), F-08 (open redirect in `track.php` click), F-10 (runtime DDL forgets `organization_id` column → first-call failure on fresh DB), F-14 (committed hex token defaults that should fail-closed), F-15 (`UPDATE campaign_sends` not org-scoped on open/click).
