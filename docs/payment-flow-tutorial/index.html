Mid-flow: production bug found

HTTP 500 · /api/billing/checkout

The next step in the script — POST to /api/billing/checkout — returned HTTP 500 with this database error:

{
  "status": 500,
  "body": {
    "error": "Database error: SQLSTATE[42S22]: Column not found: 1054
              Unknown column 'promo_code' in 'INSERT INTO'"
  }
}
Impact: Every paid plan checkout on netwebmedia.com was failing silently before this fix. Tutorial Tester is the first end-to-end test that surfaced it. The free plan signup (which doesn't touch this code path) was unaffected.

Root cause

The subscriptions table CREATE statement in bl_ensure_schema() (api-php/routes/billing.php:582) includes promo_code, discount_pct, and discount_cycles columns. But CREATE TABLE IF NOT EXISTS is a no-op when the table already exists — even if new columns have been added to the CREATE statement since the original create. The production database had the older table schema; the new columns were never backfilled.

The fix

Add three idempotent ALTER TABLE statements after the CREATE block. Each one is wrapped in try/catch with errno 1060 (Duplicate column) treated as a safe no-op — matching the pattern already used for the users.status migration directly above.

$coupon_cols = [
  ['promo_code',      "ALTER TABLE subscriptions ADD COLUMN promo_code VARCHAR(40) DEFAULT NULL AFTER mp_init_point"],
  ['discount_pct',    "ALTER TABLE subscriptions ADD COLUMN discount_pct TINYINT UNSIGNED DEFAULT 0 AFTER promo_code"],
  ['discount_cycles', "ALTER TABLE subscriptions ADD COLUMN discount_cycles TINYINT UNSIGNED DEFAULT 0 AFTER discount_pct"],
];
foreach ($coupon_cols as [$name, $sql]) {
  try { db()->exec($sql); }
  catch (PDOException $e) {
    if (strpos($e->getMessage(), '1060') === false) throw $e;
  }
}

Committed as d86ceed83 → 8d454cea3 (post-rebase). Deployed via GitHub Actions in ~3 minutes.

After fix: First request to any /api/billing/* endpoint runs bl_ensure_schema(), the ALTER TABLEs run silently, and the next checkout INSERT succeeds. No downtime, no manual DB intervention required.

Bug #2 found — pricing.html SKU mismatch

HTTP 400 · "Unknown plan"

Our first attempt to test CMO Standard used plan_code: "cmo_standard" (matching the data-plan attribute on the "Start Standard — $1,490/mo →" button at pricing.html:441). Response:

{
  "status": 400,
  "body": { "error": "Unknown plan" }
}

Every click on that button hits this 400 error. The CMO Standard $1,490/mo button is currently broken in production.

Root cause: The catalog in api-php/routes/billing.php:317-353 defines four CMO plans — cmo_starter ($249), cmo_growth ($999), cmo_scale ($2,499), cmo_enterprise ($5,999). There is no cmo_standard. There is no $1,490 CMO plan at all.

Strategic decision needed

This is not a "fix the bug" situation — it's a strategic alignment question. Two paths forward:

Option A — Update marketing

Change pricing.html button to data-plan="cmo_growth" + label "Start Growth — $999/mo →". Aligns the public site to the actual product catalog. Downside: the "$1,490 CMO Standard" price point is baked into every sales simulation, proposal template, engagement letter, and onboarding guide in docs/. Those would need updating.

Option B — Update catalog

Add a cmo_standard plan to bl_plans_seed() at $1,490 between cmo_growth and cmo_scale. Matches every existing marketing material + sales doc. This is what the marketing materials assume exists.

Recommendation: Option B. The $1,490 "CMO Standard" price point is referenced across:

  • docs/sales-management-simulation/ — Sarah Test's $17,880 year-1 deal

The catalog is the outlier. Adding cmo_standard at $1,490 brings the system into alignment with everything else.

Cancel the MP preapproval via the API (recommended)

Easiest cleanup — let the cancel endpoint do all four things at once (subscription row, user state, and MP-side cancellation). Run this from the user's logged-in session:

curl -X POST https://netwebmedia.com/api/billing/cancel \
  -H "X-Auth-Token: $TUTORIAL_TESTER_TOKEN" \
  -H "Content-Type: application/json" \
  -H "Origin: https://netwebmedia.com" \
  -d '{}'

Where $TUTORIAL_TESTER_TOKEN is the 64-char token stored in localStorage from step 6 or 8. Or simpler: just run the SQL DELETEs above and ignore the MP preapproval — it auto-expires.
