The CRM **Courses** module (`crm-vanilla/courses.html` + `js/courses.js`) is backed by **`api-php/routes/courses.php`** against the **`webmed6_nwm`** database (path-based `/api/courses`, `/api/courses/{id}/lessons`, etc.). Auth is `X-Auth-Token` for both reads AND writes (admin gated) — unlike most CRM writes which need a PHP session.

Gotchas (confirmed 2026-05-29):
- `crm-vanilla/api/schema_courses.sql` is **stale/unused** for the live feature (there is NO courses handler in `crm-vanilla/api/handlers/`). Don't reason about courses from that file — read `api-php/routes/courses.php`.
- Course **cover images** are SVGs at `crm-vanilla/assets/courses/<slug>.svg`, generated by `crm-vanilla/assets/courses/gen.js`. They deploy via `deploy-site-root.yml` (`crm-vanilla/**`). Cached `immutable` 1yr — bump the DB `cover_image` with `?v=N` to bust.
- Lessons were bulk-seeded via `_lessons/browser-inject.js` (inline `allCourses` data, POSTs to the admin lessons endpoint). The create-lesson API defaults `status` to `draft`, so seeded lessons are invisible to students unless status is set to `published`.

- Lessons **open into a reader** (`openLessonReader` in `courses.js`, added 2026-05-29): clickable rows → renders `lesson.content` (admin-authored HTML, not escaped) + prev/next + Mark Complete. `GET /api/courses/{id}` now returns `content`+`title_es`+`description_es` for lessons.
- **Known gap:** the create/update lesson handlers (`POST`/`PUT .../lessons`) do NOT persist `title_es`/`description_es` — so lesson titles are English-only in the DB even though `_lessons/*.json` seeded ES values. `lname()` falls back to the English title. There is no `content_es` column (lesson bodies are English-only).

Live as of 2026-05-29: 29 courses, 4 lessons each (116 total).
