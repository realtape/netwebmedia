## Courses module security spot-check — TWO CONCERNS

The schema and routes were spot-checked. Module is **paid-gated** via `requirePaidAccess()`. But:

**Concern #1 — Multi-tenant gap (medium severity).** `courses.php` and `schema_courses.sql` have **zero `organization_id` columns** and zero references to `tenant_where()` / `tenant_owns()`. All courses are global across tenants. That's defensible **iff** the intent is "NetWebMedia teaches all subaccounts the same curriculum," which matches the seed data (15 NWM-branded courses). But white-label resellers will see "NetWebMedia CRM Masterclass" inside their own branded CRM. Either (a) document this as intentional and add a `nwm_branded` flag for the future, or (b) add `organization_id` now to allow per-tenant course catalogs. Right now it's an architecture-debt landmine, not a security hole.

**Concern #2 — XSS hole in `crm-vanilla/js/courses.js` (HIGH severity, ship-blocker).** Six `innerHTML` writes; only ONE escapes (line 103, search query). Lines 117, 120, 121, 132, 216, 217 inject `c.name`, `c.tagline`, `data.course.name`, `data.course.icon`, and `c.color` (used inside `style="..."`) directly into the DOM. Although today these come from admin-seeded DB rows, the moment a tenant admin can create courses (which the API supports indirectly via raw INSERT), an XSS exists. **Route every interpolated DB string through `CRM_APP.esc()`** per the project rule — the file even has `escapeHtml` defined locally and ignores it. Fix before any tenant gets course-create access.

**Server-side route logic** (`api-php/routes/courses.php`) is otherwise clean: parameterized PDO throughout, no string concatenation of user input into SQL, proper enrollment ownership check on `complete-lesson`. No SQL injection vector spotted.

## 5 hidden bugs introduced today

1. **`courses.js` XSS** (above) — HIGH. Pre-prod gate before any tenant gets author rights.
2. **404 on `/assets/founder-carlos.jpg`** — every homepage hit logs a 404. SVG fallback masks it visually but Sentry will accumulate noise. Either upload the JPG (recommended — JPG is half the bytes of SVG monogram for a small avatar) or invert: SVG-first with no fallback request.
3. **`courses.js` line 103 search-empty-state innerHTML** uses `escapeHtml(q)` correctly, but line 132's `progress` value goes into a `style="width:Xpx"` attribute concatenation. Today `progress` comes from a server-side `(int)` cast so it's safe — but it's the kind of pattern that breaks silently when somebody adds a string-typed field. Refactor to `el.style.width = progress + '%'` instead of `innerHTML += '...style="width:' + progress`.
4. **Statement splitter risk in `schema_courses.sql`**: lines 87 ("…40 ad variants a week on Meta, Google, TikTok — the only way to win in 2026") use em-dashes and commas heavily. No `;` inside literals, but the migration runner's naive splitter is one fragile typo away from tearing a row in half. Already raised in CLAUDE.md — re-flag because today's changes added a lot of literal text to the migration corpus.
5. **`deploy-site-root.yml` is now responsible for `crm-vanilla/`**, but the deprecation note in CLAUDE.md says `deploy-crm.yml` is "manual fallback only". The Courses module includes both `crm-vanilla/api/schema_courses.sql` (auto-runs via migrate) and `crm-vanilla/js/courses.js`. If the site-root workflow's path filter omits one of these globs, a future Courses-only commit could deploy the JS without running the schema migration → 500s on the API. Verify path filter covers `crm-vanilla/api/schema_*.sql` AND that the migrate POST runs in this workflow (CLAUDE.md says it does, but verify the line).
